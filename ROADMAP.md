# ROADMAP: выравнивание TS-зеркала MSLang с PHP-эталоном

> Правило №0 действует везде: **PHP — единственный эталон** (точнее — спецификация поведения; числовая модель — IEEE-754/JS, см. `MSLang.md` → «Числовая модель» и `CLAUDE.md`). TS обязан повторять PHP бит-в-бит: имена полей, состав childItems, имена обработчиков, числовые значения констант, алгоритмы, тексты и коды ошибок. Отклонения допустимы только там, где их вынуждает специфика языка. Порядок правок — PHP-first; перед PR с правками `src/` — бамп версии языка.

## 2. Хвосты паритета — подтверждённые расхождения поведения

Каждый пункт подтверждён живым прогоном одного и того же скрипта в двух движках (цифры и разбор — в `FINDINGS-2026-07-02.md`). Все правки — в TS (эталон уже верен), каждая закрывается зеркальной парой тестов.

- **P1-10 (хвост): бюджет данных у `flip`/`concat`.** `keys`/`values`/`reverse` списывают бюджет одинаково (48/48/48), но `flip`: TS +0 против PHP +48 (TS наполняет пустой массив через `setProperty` без учёта); `concat([4,5])`: TS +48 против PHP +80 (учтены только начальные элементы, допушенные не считаются). Где: `src/stackvariablearray.ts:300,348`. *Готово, когда:* `getAllocatedBytes` после всех пяти методов совпал с PHP; зеркальная пара в `tests/limits.ts`/`TestLimits.php` расширена на `reverse`/`flip`/`concat`, прогон — без присваивания результата.
- **P1-19 (хвост): запись по индексу в скаляр.** Чтение готово (скаляр бросает `Cannot read offset`, float-ключ нормализуется). Запись — нет: `let n=5; n[0]=1` (а также булево, null, строка `s[0]="x"`) в PHP бросает `Cannot set offset` (`php/src/StackVariable.php:352-355`), в TS молча игнорируется (`src/interpreter.ts:1983` зовёт `setProperty` без проверки типа; базовый `setProperty` тихо выходит). Зеркальных тестов на запись в скаляр нет ни в одном наборе — добавить в оба.
- **P1-5 (хвост): последняя незеркальная ветка `callFunction`.** Переменная есть, но не `vtFunction`: PHP бросает `ContextException("Call global function " + name)` (`php/src/ContextInterpreter.php:925`), TS — базовый `MSLangException('variable "..." is not function')` (`src/contextinterpreter.ts:861-863`). Расходятся и тип, и текст; ветка достижима одинаково. Остальные 14 сайтов `ContextException` зеркальны.
- **P2-6 (хвост) = M6-4: контекст скалярных значений host-массива.** Вложенные массивы контекст получают, но скалярные ветки `#addValue` создают значения без него (`src/stackvariablearray.ts:55-68`), тогда как PHP передаёт контекст во все ветки (`php/src/StackVariableArray.php:60-68`). Следствие: строки из host-массива не попадают в бюджет песочницы. Конструкторы TS `StackVariableBoolean`/`Number`/`Object`/`Null`/`Undefined` вообще не принимают `context` — добавить по образцу PHP. Либо явно зафиксировать здесь, что учёта массивов достаточно.
- **P2-9 (хвост): `funcInvoke_indexOf` → `funcInvokeIndexOf`.** PHP-эталон — `funcInvokeIndexOf` без подчёркивания; переименовать тройку `funcInvoke_indexOf`/`...Args`/`...Return` в `src/stackvariablestring.ts:39-48` **и** в `src/stackvariablearray.ts:196-204`. Проверено экспериментально: поиск методов регистронезависимый, диспетчеризация не сломается; попутно имя `FunctionEntry` выровняется с PHP (`'indexOf'` → `'IndexOf'` — сейчас это наблюдаемое расхождение).
- **P2-8 (хвост): зеркальный тест на `invokeMethod`.** Правка кода сделана, теста из критерия готовности нет ни в одном наборе. Нюанс: PHP печатает полное имя с namespace (`DevBx\MSLang\StackVariable`), TS — короткое `StackVariable`; тесту понадобится нормализация, а лучше — решить, надо ли выравнивать текст дословно.
- **P2-13 (хвост): `FunctionParameter` default-тип.** TS: `type: VariableType = VariableType.vtUndefined` (=0), эталон PHP: `?int $_type = null` (`functionparameter.ts:12` против `Functions/FunctionParameter.php:21`). Не языковое отклонение — в TS возможно `VariableType | null = null`; дефолт реально задействуется. Заодно убрать маркер `// @TODO` на `functionparameter.ts:17`.

## 3. Инструменты и процесс — недоделанные части

- **M1-2 (хвост): `tsconfig.test.json` сломан.** `tsc -p tsconfig.test.json --noEmit` падает с 17 ошибками (NodeNext требует расширения `.js` в импортах), из-за этого не работает команда `test:compile` (`package.json:42`). Типизация для CI решена отдельным `tsconfig.typecheck.json` — поэтому: либо перевести `tsconfig.test.json` на bundler-резолв, либо осознанно удалить его вместе с `test:compile`.
- **M1-3: общий `tests/_unwrap.ts`.** Сейчас два разных `unwrap()` с разной семантикой: `fuzz.ts:80` превращает undefined→null и NaN/±Infinity→строки, `cross.ts:43` возвращает как есть. Свести в общий модуль; добавить нормализацию `-0→0` зеркально `php/tests/fuzz.php:43`.
- **M1-4 / §7-3 (хвост): грамматика фаззера.** Генератор по-прежнему выдаёт только `+ - * == != && ||` (`tests/fuzz.ts:53`, опасные операции намеренно опущены) — расширить на `/ % < >`, именно там копятся расхождения семантики. Также `test:fuzz:check` захардкодил путь `../php` вместо `MSLANG_PHP_ROOT` (`package.json:37`).
- **§7-1 (хвост): слепые зоны mirror-теста.** (а) `getFullVersion` не сверяется (только компоненты VERSION/REVISION — дрейф формат-строки не ловится); (б) нет падения при 0 совпадений регэкспа в **обоих** парных файлах: смена формата записи в двух файлах сразу даст тихо-зелёный (читатели молча вернут пустые множества, `mirror_Version` проходит при null===null). Добавить проверки непустоты.
- **§7-2 (хвост): парность тестов — по парам файлов, не по объединению.** Сейчас сверяется объединение tests+bugs ↔ Test+TestBugs, поэтому переезд теста между файлами не ловится (факт: `077_VarRedeclaresLetFails` в TS лежит в `bugs.ts`, в PHP — в `Test.php`). Пара `limits.ts`↔`TestLimits.php` из сверки исключена совсем (в TS русские описания, в PHP английские имена) и не сторожится даже по количеству — добавить хотя бы сверку числа тестов.

## 7. Процесс против дрейфа (действующие правила)

1. **Mirror-тест констант — обязательный гейт** перед каждым коммитом (`npm run test:mirror`). Известные слепые зоны — задача §7-1 в разделе 3.
2. **Тест-парность** сторожится `mirror_test_parity`; довести до пофайловой сверки — задача §7-2 в разделе 3.
3. **Фаззер — рабочий и двусторонний** (`test:fuzz:check`). Каждое пойманное расхождение фиксируется зеркальной парой `.msl`/`.expected` в обоих `tests/scripts`, а не только текстом в `known-divergences.md`. Расширение грамматики — задача M1-4 в разделе 3.
4. **Версии — PHP-first.** Номер и дата меняются сначала в `php/src/Version.php`, затем 1:1 в `version.ts` и `package.json`; бамп — командой перед PR при правках `src/`; CI требует бамп, `mirror_Version` сторожит равенство.
5. **CI на обеих сторонах** работает (TS: тесты+типизация+лимиты; PHP: composer test).
6. **Дисциплина направления.** Любая новая фича/обработчик/метод/константа — сначала в PHP, потом 1:1 в TS. Код в TS без PHP-аналога — ошибка процесса, чинится добавлением в PHP. Решения с развилкой (D-1, D-2) выносятся владельцу отдельными пунктами, а не правятся молча.

## 8. Известные вынужденные отклонения TS (не переносить в PHP и не «чинить»)

Оба закреплены комментариями «ЯЗЫКОВОЕ ОТЛИЧИЕ от PHP» прямо в коде и стражами-тестами:

- **P2-7 (`objPropHandler`, `interpreter.ts:1255`).** В TS Ref кладётся без контекста, отсутствующее свойство — обычный `StackVariableUndefined` (в PHP — записываемый Ref с контекстом). Причина: `funcEntryCache` захватил бы `Proxy(StackVariableRef)` вместе со scope — падал бы чужой вызов после выхода из короткоживущего scope (страж `Bug_FuncEntryCache_ProxyOnDeadScope`).
- **P2-12 (`callFunction`, `contextinterpreter.ts:879`).** Перебор аргументов через `Math.max` вместо строгого цикла по `getParameters()`: встроенные вариативные функции TS недообъявляют параметры. Для пользовательских функций разницы нет. Снимается долгом «вариативные builtin как rest» (раздел 5).
