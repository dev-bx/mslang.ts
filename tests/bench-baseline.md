# Бенчмарк — точка отсчёта перед Тиром 4 (TS и PHP)

Снято **2026-06-14**, до оптимизаций Тира 4 (O-1..O-13).
Среда: Node v24.15.0, PHP 8.5.4 (NTS, opcache), Linux.
Прогон: `npm run bench` / `php tests/bench.php` (прогрев 100 итераций, затем замер).
Цифры — среднее по 3 прогонам, разброс ~1–3%. Сценарии в `bench.ts` и `bench.php` совпадают 1-в-1.

| Сценарий | runs | TS ms/run | PHP ms/run | TS быстрее |
|---|---:|---:|---:|---:|
| parse (arithmetic) | 1000 | 0.0070 | 0.368 | **53×** |
| parse (hot loop) | 1000 | 0.0122 | 0.665 | **55×** |
| exec (arithmetic) | 1000 | 0.0325 | 1.118 | **34×** |
| exec (complex bool) | 1000 | 0.0279 | 1.229 | **44×** |
| **exec (hot loop 1000×)** | 50 | **11.295** | **167.99** | **14.9×** |
| exec (string concat 100×) | 50 | 4.097 | 18.75 | 4.6× |
| **exec (array push 200×)** | 50 | **9.475** | **45.77** | 4.8× |
| exec only (arithmetic, parsed once) | 1000 | 0.0209 | 0.757 | 36× |
| exec only (hot loop 1000×, parsed once) | 50 | 11.365 | 166.72 | 14.7× |

## Что показывает база

- **TS уже кратно быстрее PHP во всех сценариях** (V8 JIT против интерпретатора PHP). Поэтому Тир 4 —
  это про **абсолютную скорость TS в браузере**, а не про «догнать PHP» (TS давно впереди). Слово
  «снижает дрейф с PHP» в ROADMAP про O-1/O-3 — это о **форме кода** (методы на прототипе, без Proxy),
  а не о производительности.
- **Где у TS наименьшее преимущество — там и жир Тира 4.** На чистой арифметике/парсинге TS быстрее
  в 34–55×, а на горячих циклах с мутацией и метод-вызовами — лишь в 4.6–4.8× (string concat, array
  push) и 14.9× (hot loop). Значит именно накладные расходы цикла (Proxy у `StackVariableRef`,
  копирование scope, поля-стрелки) съедают преимущество — это цели O-1/O-2/O-3.
- **Главные цели (TS):** `exec (hot loop 1000×) ≈ 11.3 мс` и `exec (array push 200×) ≈ 9.5 мс`.
- `parse` дёшев и не доминирует (`exec only ≈ exec` для hot loop) — оптимизируем рантайм, не парсер.

## Как сравнивать после Тира 4

`npm run bench` (и при желании `php tests/bench.php`) тем же образом — 3 прогона, среднее, та же
машина/версии. Прирост считать в % к базовым **TS** ms/run по `exec (hot loop 1000×)` и
`exec (array push 200×)`. PHP-колонка — для контекста (Тир 4 PHP не трогает, его цифры не должны меняться).

## Результаты Тира 4 (2026-06-14)

Сделано (безопасные, проверенные правки; PHP не трогался):
- **O-1 (шаг 1)** — общий обработчик Ref-прокси вместо 6 замыканий на каждое чтение переменной.
- **O-4** — регистрация обработчиков узлов через `.bind(this)` вместо стрелок-обёрток.
- **O-3** — общий объект `properties` вместо литерала на каждый экземпляр (аллокации, не CPU).

| Сценарий | TS до | TS после | Δ |
|---|---:|---:|---:|
| exec (hot loop 1000×) | 11.295 | ~6.6 | **−42%** |
| exec (array push 200×) | 9.475 | ~8.49 | −10% |
| exec (string concat 100×) | 4.097 | ~3.73 | −9% |
| exec (arithmetic) | 0.0325 | ~0.031 | ≈0 |
| parse (*) | 0.007–0.012 | без изменений | — |

Главный выигрыш — общий Ref-прокси-обработчик: `getProxy()` аллоцировал свежий объект с шестью
трапами-замыканиями на КАЖДОЕ чтение переменной (а горячий цикл читает по нескольку переменных на
итерацию). Один общий обработчик убрал это — −42% на горячем цикле.

**Намеренно НЕ сделано (высокий риск рерайта при умеренной выгоде, нужен отдельный разбор):**
- **Полный O-1** — заменить Proxy на явные методы-делегаты у `StackVariableRef` (как `__get/__call`
  в PHP). Шаг 1 уже снял основную стоимость Proxy (аллокацию обработчика); остаток — вызовы трапов
  (~2.4% по профилю). Полное удаление Proxy — размашистая правка ядра, выгода ~5%, риск высокий.
- **O-2 (COW `_variables`)** — `pushExecutionStack` копирует все переменные `Object.assign` на каждый
  блок; логика «копирования назад» в `popExecutionStack` тонкая и критична для семантики scope. ~6%
  при высоком риске. Нужен отдельный, тщательно протестированный рерайт.
- O-5..O-13 — парсер/лексер и мелочи: профиль показал, что парсинг НЕ доминирует (`exec only ≈ exec`),
  поэтому низкая отдача. O-13 (`keep_classnames`) — предусловие минификации, не выигрыш сам по себе.
