# Бенчмарк — точка отсчёта перед Тиром 4

Снято **2026-06-14**, до оптимизаций Тира 4 (O-1..O-13).
Среда: Node v24.15.0, Linux. Прогон: `npm run bench` (прогрев 100 итераций, затем замер).
Цифры — среднее по 3 прогонам, разброс ~1–3%.

| Сценарий | runs | ms/run (база) |
|---|---:|---:|
| parse (arithmetic) | 1000 | 0.0070 |
| parse (hot loop) | 1000 | 0.0122 |
| exec (arithmetic) | 1000 | 0.0325 |
| exec (complex bool) | 1000 | 0.0279 |
| **exec (hot loop 1000×)** | 50 | **11.295** |
| exec (string concat 100×) | 50 | 4.097 |
| **exec (array push 200×)** | 50 | **9.475** |
| exec only (arithmetic, parsed once) | 1000 | 0.0209 |
| exec only (hot loop 1000×, parsed once) | 50 | 11.365 |

## Где жир (главные цели Тира 4)

- **exec (hot loop 1000×) ≈ 11.3 мс** — самый горячий путь. На каждой итерации `a = a + i` и `i++`
  крутят `StackVariableRef`/`Proxy`, копирование scope, поиск переменных. Цель O-1 (убрать Proxy,
  ожидаемо ~20%), O-2 (COW `_variables`), O-3 (поля-стрелки → прототип).
- **exec (array push 200×) ≈ 9.5 мс** — второй по тяжести. Метод-вызовы + рост Map + бюджет данных.
- `parse` дёшев (0.007–0.012 мс) — парсер не цель оптимизации.
- `exec only` vs `exec` для hot loop почти не отличается (11.37 vs 11.30) — парсинг не доминирует,
  всё время в интерпретаторе. Это подтверждает: оптимизировать надо рантайм, не парсер.

## Как сравнивать после Тира 4

`npm run bench` тем же образом (3 прогона, среднее), та же машина/версия Node. Смотреть прежде всего
на `exec (hot loop 1000×)` и `exec (array push 200×)`. Прирост считать в % к этим базовым ms/run.
