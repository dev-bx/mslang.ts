# Расхождения TS ↔ PHP

Найденные фаззером (`tests/fuzz.ts` + `tests/fuzz.php`) расхождения и их статус. **PHP — эталон.**

## Закрытые

Класс расхождений вокруг loose equality `==`:

- `false != null` теперь `false` (как в PHP), было `true` в TS.
- `null == 0`, `null == ""` теперь `true` (как в PHP).
- `NaN == true` теперь `true` (PHP cast NaN→bool даёт true).
- `("b" - "1") == true` теперь `true` (NaN == true → true).
- `0 == ""` теперь `false` (PHP 8 поведение, было `true` в TS).
- `null == "0"`, `undefined == "0"` теперь `false` (как PHP и V8), было `true` в TS.
  Причина: `null` против строки PHP сравнивает КАК СТРОКИ (`null` → `""`), поэтому
  `null == ""` истина, а `null == "0"` ложь. TS-зеркало же шло коротким путём
  `!toBool(строка)`, а `toBool("0")` ложь — отсюда ошибочный `true`. Нашёл фаззер
  (seed 11) на выражении `((true && "") + (1 * null)) == null`. Починка — ветка
  «null/undefined против строки» в `phpLooseEqual`.

Решено модулем `src/phpsemantics.ts`: функция `phpLooseEqual` повторяет PHP 8
loose equality, используется в `StackVariable.compare` и `StackVariableNumber.compare`.

Регрессии стерегут cross-runtime скрипты `tests/scripts/014..020` и `051`, а также
зеркальная пара тестов `089_LooseEqualNullString`.

### Истинность (truthiness) — единая точка по JS

Раньше оба движка одинаково, но НЕ по JS, считали истинность в условиях: пустой
массив `[]` был ложью, а `NaN` в `if`/`while` — истиной. Теперь истинность единая
и ровно как в JS: ложь только у `0`, `-0`, `NaN`, `""`, `null`, `undefined`,
`false`; всё прочее (любой массив, объект, строка `"0"`, дата, хост-объект) —
истина. Считает её одна точка `Interpreter.isTruthy`, через неё проходят `!x`,
`if`/`while`/`for`, `?:`, `&&`/`||` и колбэки `filter`/`find`/`some`/`every` —
поэтому одно значение даёт один результат во всех контекстах. Дубль той же логики
живёт в `castAs(vtBoolean)` у числа/массива/даты (для явного `(bool)`/`!!`).

Важно: это НЕ трогает loose equality `==`. Там осталась PHP-семантика (`NaN == true`
→ true, `[] == false` → true, `"0" == false` → true) — её держит отдельный
`phpLooseEqual`/`phpToBool`. Две истинности живут раздельно специально, регрессии
`==` по-прежнему стерегут скрипты `014..020`.

### Имя типа строки в тексте ошибок

`getTypeName()`/`typeName` для значения-строки давал разное имя на двух движках:
в `switch` не было ветки для `vtString`, и обе стороны проваливались в запасной
путь. PHP через `array_search` по константам возвращал имя константы — `vtString`,
а TS-зеркало срезало приставку и опускало регистр — `string`. Расхождение видно
в тексте ошибок, где имя типа подставляется в сообщение, например у `instanceof`:
`var a = 5; var b = "x"; a instanceof b` → PHP «…got vtString», TS «…got string».

Починка — добавили в `switch` ветку `vtString → 'string'` сначала в PHP-эталоне,
затем в TS. Теперь обе стороны дают чистое `string`, как и все прочие типы
(`number`, `bool`, `array`, `object`…). Регрессию стережёт cross-runtime скрипт
`tests/scripts/052-instanceof-typename` (ловит текст ошибки через `catch`) и
зеркальная пара тестов `090_StringTypeName`.

### Число → строка

Раньше PHP печатал число родным способом и расходился с JS: дробь резалась по `precision`
из php.ini (`0.1+0.2 → "0.3"`), верхний `E` и неверные пороги (`1e21 → "1.0E+21"`,
`0.000001 → "1.0E-6"`), `-Infinity` терял знак (`"Infinity"`), а `Infinity` внутри массива
печатался как `"INF"`.

Решено единым форматтером `StackVariableNumber::numberToJsString` — печать ровно как JS
`Number.toString` (кратчайшая точная запись, пороги экспоненты `1e21`/`1e-6`, нижний `e`,
`-0 → "0"`). Подключён ко всем путям печати числа: `castAs(vtString)` (а значит и склейка
через `+`), и стрингификация массива — `join()` и `toPrimitive()`. В TS это нативный
`toString`/`Array.join`; правка касалась PHP-эталона, кроме `-Infinity` (его теряли оба
движка). Регрессии стерегут cross-runtime скрипты `tests/scripts/039-number-to-string` и
`040-array-number-to-string` плюс trap-тест `P0_21_NumberToString`.

Этот форматтер — единая точка печати числа: будущие `JSON.stringify`, `toFixed`, сортировка
по умолчанию и `Number.format` обязаны звать его же, а не родной вывод.

### JSON (`JSON.parse` / `JSON.stringify`)

Разбор идёт нативными парсерами (`json_decode` в PHP, `JSON.parse` в TS) — они согласованы по
значениям и тому, что считают валидным. `stringify` собираем строку ВРУЧНУЮ на обоих движках
одним алгоритмом (не `json_encode`/нативный `JSON.stringify`): числа — через `numberToJsString`,
`NaN`/`Infinity` → `null`, экранируем только `"`/`\`/управляющие, не-ASCII печатаем сырым,
`undefined` в массиве → `null`/у ключа выбрасывается. Порядок ключей объекта приведён к
JS-каноничному (целые индексы по возрастанию, затем по вставке), поэтому совпадает и между
движками, и с нативным `JSON.stringify`. Проверено дифф-фаззером (parse→stringify против V8) и
скриптом `tests/scripts/041-json`.

Известный край: `json_decode` в PHP по умолчанию ограничивает вложенность 512 уровнями, а
`JSON.parse` в JS — нет. JSON глубже 512 уровней: PHP вернёт `default`/`undefined`, TS разберёт.
На практике такие данные патологические.

## Как искать новые

```bash
# TS-сторона генерирует скрипты + свои результаты
npx tsx tests/fuzz.ts --count 1000 --seed 1 > /tmp/fuzz.jsonl

# PHP-сторона прогоняет тот же набор и сверяет с TS
php ../php/tests/fuzz.php /tmp/fuzz.jsonl
```

Меняй `--seed` — найдёшь новые входы. После починки бага добавь сценарий в
`tests/scripts/` как `*.msl` + `*.expected`, чтобы он попал в обязательный набор
и больше не вернулся.
