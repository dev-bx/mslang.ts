# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## ⚠️ Правило №0: PHP — эталон проекта

**Единственный эталон MSLang — PHP-реализация в соседнем репозитории `mslang.php` (`/work_space/Bitrix.DevBx/devbx.mslang/php/src/`).** TypeScript в этом репозитории (`mslang.ts`) — это **зеркало**, и больше ничего.

Это правило не обсуждается и не подменяется в рамках задачи. Оно действует ВСЕГДА.

**Перед любой правкой** (фича, обработчик узла, метод на `StackVariable*`, константа, алгоритм, имя поля, форма хранения) проверь себя:
1. Эта правка уже сделана в PHP? — Если нет, сначала идёшь в PHP и делаешь там. Только потом 1:1 переносишь в TS.
2. Я меняю TS, потому что в PHP так? — Если нет, STOP. Никогда не правь PHP «под TS» и никогда не оправдывай это словами «TS-эталон», «TS уже сделал», «TS чище» — это запрещено.
3. Если в TS уже есть код, которого нет в PHP, — это **ошибка процесса**, исправляется добавлением в PHP, а не переносом TS-формы как нового эталона.

Отклонения от PHP-формы допустимы **только** там, где их вынуждает специфика языка (`mb_strlen` против `string.length`, `array_slice` против `Array.prototype.slice`, отсутствие `null` против `undefined` и т.п.). Тип данных, имена полей, состав childItems, имена обработчиков, нумерация констант — всё это **обязано совпадать** с PHP бит-в-бит.

Подробный протокол — в разделе «Зеркальная разработка с PHP» ниже. Зеркальный тест констант (`npm run test:mirror`) обязан быть зелёным.

### Уточнение от 2026-06-13: эталон — спецификация, а не «PHP всегда прав»

Раньше правило читалось как «PHP непогрешим, TS слепо копирует». Теперь точнее:

- **Эталон — спецификация поведения языка.** Сама PHP-реализация тоже может содержать баг относительно спецификации.
- **Числовая модель MSLang — IEEE-754 / JavaScript** (решение владельца): деление на ноль даёт `±Infinity`/`NaN` (не общий `INF`); остаток `%` вещественный (`(7/2)%2 → 1.5`, `7%0 → NaN`); `Math.round` округляет половины к +∞ (`round(-2.5) → -2`); строка→число по правилам JS `Number()` (`"0x1A" → 26`); две числовые строки в `==` сравниваются как строки (`"1"=="01" → false`); битовые `& | ^` и сдвиги — 64-битные через BigInt в обоих движках. Подробности — в `MSLang.md` → «Числовая модель».
- **PHP-first остаётся, но как порядок багфиксов.** Если реализация расходится со спецификацией — чинится сторона, что отступила. Когда отступает PHP (а TS уже верен по спецификации) — PHP исправляется **первым** как баг, затем сверяется/зеркалится в TS. Это не «правка PHP под TS», а исправление PHP по спецификации.
- **Что не изменилось:** числовые значения констант, имена обработчиков, состав childItems, форма хранения — обязаны совпадать бит-в-бит; зеркальный тест констант (`npm run test:mirror`) — жёсткий гейт; порядок правок — PHP-first.
- **Каждое исправление P0 — с зеркальной парой тестов** в `tests/*.ts` и `php/tests/*.php` под одинаковыми именами (матрица P0 — в `ROADMAP.md`).

Направление правки по каждому P0 (где правится PHP, где TS, где обе стороны) — в `ROADMAP.md`, раздел «P0».

## Стиль русского языка

Пиши простым, обыденным русским языком. Избегай профессионального жаргона и заимствованных терминов там, где смысл можно передать обычными словами.

### Запрещённые/нежелательные слова и замены

Не используй без крайней необходимости:
- «идемпотентность», «идемпотентный» → «повторный вызов безопасен», «повторные вызовы дают тот же результат», «вызвать дважды безопасно»
- «дедуплицирует» → «отсеивает дубли», «убирает повторы», «игнорирует повторы»
- «валидирует» → «проверяет»
- «инвалидирует» → «сбрасывает», «помечает недействительным»
- «персистентность» → «сохраняется», «не теряется между запусками»
- «консистентность» → «согласованность», «данные не расходятся»
- «мутирует» → «изменяет»
- «инстанцировать» → «создать»

### Принципы
1. Термин допустим, только если у него нет короткой замены и читатель-разработчик его точно знает (например, «транзакция», «кэш», «индекс» — оставляй).
2. Если есть выбор между заимствованным термином и описанием в 3–5 слов — выбирай описание.
3. После тире давай не синоним-термин, а пояснение «по-человечески»: что именно это означает на практике.

### Пример

❌ Плохо:
> `addPoolItem` дедуплицирует по `externalId` — идемпотентность подтверждена в коде.

✅ Хорошо:
> `addPoolItem` отсеивает дубли по `externalId` — повторные вызовы с тем же значением безопасны, это видно по коду.

## О проекте

`devbx.mslang` — это **MSLang**, небольшой встраиваемый язык скриптов, написанный на TypeScript. Собирается в одиночный UMD-файл без внешних зависимостей (`dist/mslang.umd.js`, глобальное имя `DevBX.MSLang`) и предназначен для запуска скриптов в браузере в рамках Bitrix DevBx. Комментарии в исходниках — на русском.

## Зеркальная разработка с PHP

У MSLang есть **зеркальная PHP-реализация** в соседнем репозитории, такая же по структуре:

- TypeScript: `/work_space/Bitrix.DevBx/devbx.mslang/ts/src/`
- PHP: `/work_space/Bitrix.DevBx/devbx.mslang/php/src/`

**Оба репозитория приватные** (`dev-bx/mslang.php`, `dev-bx/mslang.ts`). Поэтому:
- не предлагать публичные ссылки «зайди и посмотри» — это 404 для всех вне команды;
- не закладывать в публичный GitHub Actions клонирование соседнего репо — токен по умолчанию не имеет доступа;
- mirror-test (`npm run test:mirror`) на CI скипается без `MSLANG_PHP_ROOT`, и это нормальное поведение, локально он запускается;
- любые предложения про cross-repo CI/публичный NPM сначала обсуждать с пользователем — настройка приватных доступов это отдельный разговор.

Имена классов (`CodeLexer`, `CodeParser`, `Interpreter`, `ContextInterpreter`, все `StackVariable*`, `MathFunctions`, `FunctionEntry`, `FunctionParameter`, типы `VariableType` / `NodeType` / `LexerType` / `CompareType` / `InterpreterNodeType` / `ContextType`) совпадают один-в-один.

**Правила, обязательные для любых правок:**

1. **PHP — эталон.** Любая новая реализация (фича, обработчик узла, метод на `StackVariable*`, константа, алгоритм) делается **сначала в PHP**, затем переносится в TS **один-в-один**. Отклонения допустимы только там, где их вынуждает специфика языка (например, `mb_strlen` против `string.length`, `array_slice` против `slice`). Если в TS появилось что-то, чего нет в PHP, — это ошибка процесса, надо сначала добавить в PHP.

2. **Числовые значения констант (`VariableType`, `LexerType`, `NodeType`, `CompareType`, `InterpreterNodeType`, `ContextType`) должны совпадать в обеих реализациях бит-в-бит.** Отклонения в нумерации недопустимы — один и тот же исходник MSLang обязан работать в обоих интерпретаторах с одинаковыми результатами.

3. Любая правка в PHP тянет за собой такую же правку в TS:
    - Добавил/удалил константу — синхронизируй в обоих файлах.
    - Добавил/удалил обработчик/метод — добавь зеркало в TS.
    - Изменил алгоритм (например, `plusHandler`, `toPrimitive`) — сверь поведение, а не только сигнатуры.

4. **Зеркальный тест констант** обязан проходить перед коммитом: `npm run test:mirror`. Он читает оба набора файлов и валится при первом же расхождении в значениях или именах.

5. Тесты тоже зеркальные: TS — `tests/tests.ts`, PHP — `tests/Test.php`; пусть имена и набор тестов совпадают.

## Как выполняется скрипт

Скрипт проходит три шага. Готовый пример — функция `createCodeContext()` в `tests/tests.ts`:

1. **Лексер** (`lexer.ts`) — `CodeLexer` разбивает исходный текст на токены. Виды токенов лежат в числовой таблице `LexerType`. `LexerTypeArray` — это `number[]` с дополнениями, через него в парсер передают набор ожидаемых видов токенов.
2. **Парсер** (`parser.ts`) — `CodeParser` читает токены лексера и строит плоский `ParseNode[]` (у каждого узла могут быть дочерние узлы). Точка входа — `parseCode(nodeList, getFirstToken, inline, endLineTypes)`, она вызывает сама себя. Виды узлов лежат в числовой таблице `NodeType`.
3. **Интерпретатор** (`interpreter.ts`) — сначала `new Interpreter()`, затем `registerHandlers()` ставит по одному обработчику на каждый `NodeType`. `ContextInterpreter(nodeList, interpreter)` хранит всё состояние выполнения; вызови `registerConst()`, чтобы добавить встроенные значения, затем `exec(true)` — выполнить и вернуть верхнее значение стека.

Интерпретатор работает как **стековая машина**, а не обходит дерево. Обработчик повторно кладёт узел в работу со служебным видом `*Finish` (таблица `InterpreterNodeType`, коды от 1000) — так при втором заходе он подбирает результаты вычисления дочерних узлов. `InterpreterNode.typeName` ищет имя и в `NodeType`, и в `InterpreterNodeType`.

## Главные понятия

- **`ContextInterpreter`** держит `_variables`, `_functions`, стек значений `_stackVars` и `_executionStack`. `pushExecutionStack()` / `popExecutionStack()` дают область видимости для блоков, циклов, вызовов функций — при возврате значения непостоянных переменных копируются обратно. Поле `_type` (`ContextType`) задаёт ход выполнения, например `ctReturn` и `ctAllowBreak`. Отдельно живут `_currentThis`, `_currentMethodOwner`, `_isCtorTDZ` (для классов и `super`, см. ниже).
- **Иерархия StackVariable** — каждое значение во время выполнения это потомок `StackVariable` (`StackVariableNumber`, `String`, `Boolean`, `Array`, `Object`, `Function`, `UserFunction`, `Class`, `Null`, `Undefined`, `DateTime`, `Ref`). Создавать их удобно через `ContextInterpreter.createVariable(type, value)`. `StackVariableRef` оборачивает значение в Proxy для передачи параметра по ссылке.
- **`VariableType`** — учти, что `vtInteger`, `vtFloat`, `vtNumber` намеренно равны одному значению `3`; типа со значением `4` нет.
- **Связь с хост-кодом** — чтобы отдать в скрипт объект хоста, наследуйся от `StackVariable` (тип `vtObject`) и реализуй `getProperty(name)` (примеры `FieldsObject`/`TestObject` в тестах). Чтобы отдать функцию хоста — опиши её через `FunctionEntry` и `FunctionParameter`, оберни в `StackVariableFunction` и зарегистрируй через `setVariable`. `registerConst()` добавляет `Math`, `DateTime`, `Error` (как пользовательский `class Error` — см. `registerErrorClass`), `debug` и постоянные значения `true`/`false`/`null`/`undefined`/`NaN`/`Infinity`.
- **Классы (`class Name [extends Parent] { constructor(...) {} method(...) {} }`)** — `StackVariableClass` хранит конструктор и словарь методов; `_parentName` — lazy-резолв родителя через `getParent(context)`, защита от циклов `PARENT_CHAIN_LIMIT=100`. Объявления классов поднимаются `hoistFunctions` наряду с user-функциями. Внутри метода/конструктора в кадре `_executionStack` (и `pushFunctionScope`) выставляются `_currentThis` = instance и `_currentMethodOwner` = класс-объявитель кода (важно: владелец кода, а не `this.getClass()` — иначе `super` зациклится у потомков). У ctor extends-класса есть **TDZ**: до первого `super(...)` обращение к `this` — ошибка. `super(...)` сбрасывает TDZ-флаг. `super.method(...)` берёт тело родителя, но `this` остаётся текущим (JS-семантика). `instanceof` ходит по цепочке `getParent` и сравнивает классы по ссылке.
- **Function-конструктор** — обычная `StackVariableUserFunction`, вызванная через `new`. Класс-обёртка создаётся лениво на самой функции (`getOrCreateWrapperClass()`), кешируется — поэтому `a instanceof Foo` и `b instanceof Foo` сравниваются по тому же классу. Методы такого «класса» цепляются как свойства instance (`this.greet = function() { ... };`); `obj.method(args)` находит их через `getProperty` и вызывает с `this = obj`.
- **Встраивание в текст** — символы `{` и `}` лексер распознаёт как `ltStartCode` / `ltEndCode`, поэтому код MSLang можно вставлять внутрь обычного текста.

## Возможности языка

- Управление потоком: `if`/`else` (и `ntNegativeIf`), `switch`/`case`/`default`/`break` (с проваливанием), `for`, `while`, `continue`, `break`, `return`, инкремент/декремент.
- Выражения: арифметика и побитовое И, сравнения и `&&`/`||`, `instanceof`.
- Данные: массивы (push, индексы, ассоциативные ключи, распаковка), объекты, пользовательские функции с замыканиями, function-выражения (`x = function() { ... }`).
- Объектная модель: `class Name [extends Parent] { constructor(...) {} method(...) {} }`, `new Name(args)`, `this`, `super(...)`, `super.method(...)`, `instanceof`, авто-вызов родительского ctor для extends-класса без своего конструктора.
- Старый JS-стиль: `function Foo(args) { this.x = ...; this.method = function() {...}; }` + `new Foo(...)` — function-конструктор с методами на instance.
- Ошибки: `try`/`catch`/`finally`, `throw`, `new Error(...)`, пользовательские иерархии через `class MyError extends Error { ... }`.
- Вызовы: пространство имён (`Class::fn()`), метод объекта (`obj.fn()`), функция-выражение в свойстве.
