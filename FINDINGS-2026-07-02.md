# Находки проверки от 2026-07-02

Побочный результат актуализации `ROADMAP.md`: каждый пункт прежней карты сверили с фактическим кодом обоих репозиториев (ветка `feature/data-features`, TS `7f389ee` / PHP `fe86b5e`, язык 2.1.0). Спорные вердикты перепроверялись независимым вторым проходом; ключевые расхождения подтверждены живыми прогонами одинаковых скриптов в двух движках. Здесь собрано то, что нашлось попутно: настоящие расхождения поведения, слепые зоны инструментов и риски. Задачи из этого списка уже внесены в `ROADMAP.md` (разделы 2–5) — тут подробности и доказательства.

## 1. Подтверждённые расхождения поведения (TS против PHP-эталона)

Каждое воспроизведено прогоном одного и того же скрипта MSLang в обоих движках.

### 1.1. Бюджет данных: `flip` и `concat` списывают не как эталон (→ ROADMAP P1-10)
На базе `let a = [1,2,3]` дельта `getAllocatedBytes`:

| Метод | PHP (эталон) | TS |
|---|---|---|
| `a.keys()` / `a.values()` / `a.reverse()` | +48 / +48 / +48 | +48 / +48 / +48 ✓ |
| `a.flip()` | **+48** | **+0** |
| `a.concat([4,5])` | **+80** | **+48** |

Причина: PHP-методы возвращают готовый массив, `invokeMethod` оборачивает его через `createVariable`, и конструктор списывает итоговый размер (`php/src/StackVariable.php:200`). TS `flip` стартует с пустого массива (0×16) и наполняет через `setProperty` без учёта; TS `concat` списывает только начальные элементы, допушенные через `funcInvoke_push` не считаются (`src/stackvariablearray.ts:300,348`). Существующая зеркальная пара тестов покрывает только `keys`/`values` и присваивает результат переменной.

### 1.2. Запись по индексу в скаляр молча игнорируется (→ ROADMAP P1-19)
`let n=5; n[0]=1`, `let t=true; t[0]=1`, `let x=null; x[0]=1`, `let s="abc"; s[0]="x"` — PHP во всех четырёх случаях бросает `Cannot set offset` (`php/src/StackVariable.php:352-355` через `Interpreter.php:1785`), TS во всех четырёх молча ничего не делает: `src/interpreter.ts:1983` зовёт `setProperty` без проверки типа приёмника, а базовый `setProperty` (`src/stackvariable.ts:140-148`) тихо выходит. Строки `Cannot set offset` в ts/src нет нигде. Путь чтения при этом зеркален (`Cannot read offset`, нормализация float-ключа — совпадают). Тестов на запись в скаляр нет ни в одном наборе.

### 1.3. `callFunction`: последняя незеркальная ветка исключения (→ ROADMAP P1-5)
Вызов по имени переменной, которая существует, но не функция: PHP бросает `ContextException("Call global function " + name)` (`php/src/ContextInterpreter.php:925`), TS — базовый `MSLangException('variable "..." is not function')` (`src/contextinterpreter.ts:861-863`). Расходятся и тип, и текст; ветка достижима одинаково, это не языковой страж. Остальные 14 обычных сайтов `ContextException` зеркальны по типу и тексту.

### 1.4. Бюджет песочницы не видит скаляры host-массива (→ ROADMAP P2-6)
Скалярные ветки `#addValue` создают значения без контекста (`src/stackvariablearray.ts:55` строка, `:58` булево, `:62` число, `:65` объект, `:68` undefined), тогда как PHP передаёт контекст во все ветки через фабрики (`php/src/StackVariableArray.php:60-68`). Практическое следствие: строка из host-массива в PHP списывает бюджет (`php/src/StackVariableString.php:26`), в TS — нет, хотя конструктор `StackVariableString` контекст уже принимает и учёл бы. Глубже: конструкторы TS `StackVariableBoolean`/`Number`/`Object`/`Null`/`Undefined` вообще не имеют параметра `context` — даже «сделанная» null-ветка создаёт значение без него.

### 1.5. Имя `FunctionEntry`: `'indexOf'` против `'IndexOf'` (→ ROADMAP P2-9)
TS держит `funcInvoke_indexOf` (с подчёркиванием) в `src/stackvariablestring.ts:39-48` **и** `src/stackvariablearray.ts:196-204`; эталон — `funcInvokeIndexOf` (`php/src/StackVariableString.php:114`, `php/src/StackVariableArray.php:120`). Наблюдаемое следствие: имя `FunctionEntry` в TS = `'indexOf'`, в PHP = `'IndexOf'` (проверено обоими движками). Важное опровержение: переименование НЕ сломает диспетчеризацию — проверено экспериментально, поиск метода регистронезависимый (`src/stackvariable.ts:284`, `name.toLowerCase()`), а `'funcInvoke_'+name` на `:174` работает только при восстановлении имени из уже найденного ключа.

### 1.6. Тексты ошибки `invokeMethod` не дословные (→ ROADMAP P2-8)
PHP `StackVariable::class` даёт полное имя `DevBx\MSLang\StackVariable`, TS `StackVariable.name` — короткое `StackVariable`. Оба содержат «StackVariable» (критерий пункта выполнен), но бит-в-бит тексты не совпадают; зеркальному тесту понадобится нормализация или отдельное решение о выравнивании.

## 2. Слепые зоны инструментов проверки

### 2.1. `test:compile` сломан (→ ROADMAP M1-2)
`tsc -p tsconfig.test.json --noEmit` падает с exit 2: 17 ошибок (TS2834/TS2835 «нужны расширения .js в импортах при NodeNext» во всех 6 тестовых файлах; TS18046/TS7006 в `tests/cross.ts:51-61`, `tests/fuzz.ts:87-94`). Команда `test:compile` (`package.json:42`) использует этот конфиг вторым шагом и сейчас не работает. CI не страдает — типизация там идёт через отдельный `tsconfig.typecheck.json`.

### 2.2. Mirror-тест может стать тихо-зелёным (→ ROADMAP §7-1)
- Нет защиты «регэксп нашёл 0 в обоих парных файлах»: читатели `readPhpHandlers`/`readTsHandlers`/funcInvoke молча вернут пустые множества, diff пуст — зелёный (`tests/mirror.ts:136-164, 177-229`); `readPhpConsts` молча вернёт `{}` (`:29-36`).
- `mirror_Version` проходит при `null === null` (`tests/mirror.ts:234-236, 243-244`) — если формат записи версии сменится в обоих файлах, сверка исчезнет незаметно. Сценарий реалистичен: по правилу №0 формат меняют в двух файлах сразу.
- `getFullVersion` не сверяется нигде (только компоненты VERSION/REVISION) — дрейф формат-строки не ловится.

### 2.3. Парность тестов сверяется по объединению, а не по парам файлов (→ ROADMAP §7-2)
Переезд теста между файлами не ловится: `077_VarRedeclaresLetFails` в TS лежит в `tests/bugs.ts:856`, в PHP — в `tests/Test.php:3146`; пофайловый счёт расходится (211/212 и 100/99) при зелёном статусе. Пара `limits.ts`↔`TestLimits.php` исключена из сверки сознательно (комментарий `tests/mirror.ts:249-251`: русские описания против английских имён) и не сторожится даже по количеству (сейчас 11/11 — совпадение держится на дисциплине).

### 2.4. Фаззер намеренно слеп к самым опасным операциям (→ ROADMAP M1-4)
Генератор выдаёт только `+ - * == != && ||` (`tests/fuzz.ts:53`, комментарий «опускаем потенциально расходящиеся операции»). Именно на `/ % < >` копятся расхождения семантики, ради ловли которых фаззер и существует. Плюс два раздельных `unwrap()` с разной семантикой (`fuzz.ts:80` против `cross.ts:43`; undefined и NaN/±Infinity обрабатываются по-разному), нормализация `-0→0` есть только на PHP-стороне (`php/tests/fuzz.php:43`). Путь к PHP захардкожен как `../php` в `test:fuzz:check` (`package.json:37`), переменную `MSLANG_PHP_ROOT` читает только `mirror.ts`.

## 3. Риски и заметки

- **Риск минификации шире, чем помечено в конфиге.** `funcEntryCache` ключуется по `this.constructor.name` в **двух** местах: геттер `functions` (`src/stackvariable.ts:119-127`) и поиск функции по имени (`:276-284`). Комментарий-сторож у `minify:false` в `vite.config.ts` упоминает проблему, но включение минификации без `keep_classnames`/`keepNames` отдаст чужую таблицу методов. Старая цель размера «~135 КБ» устарела: неминифицированный UMD вырос до 389.6 КБ (исходник был 306 КБ на момент постановки).
- **Токен `=>` уже занят.** Лексер обеих сторон выдаёт для `=>` токен `ltArraySeparator` (38; `src/lexer.ts:47,546`, `php/src/CodeLexer.php:328`); парсер принимает его только в ветке `ntArrayPush`. Для стрелочных функций (F-1) придётся либо переиспользовать токен в новых контекстах, либо различать по контексту — это дизайн-решение, которого не было в исходной формулировке F-1.
- **O-8 классифицирован неточно.** Решение «низкая отдача, парсер не доминирует» (`tests/bench-baseline.md:65-66`) накрывает O-8 по номеру, но не по сути: O-8 — интерпретаторный горячий путь (новый `StackVariableBoolean` на каждое сравнение — 10 точек в `interpreter.ts`), а не парсерный. Если возвращаться к оптимизациям — начинать с него.
- **`LexerTypeArray` разошёлся по форме.** PHP уже имеет отдельный класс `php/src/LexerTypeArray.php` (ArrayAccess+Iterator+Countable, `hasValue()` на `:43`), TS всё ещё `extends Array<number>` без `hasValue` — по правилу №0 TS должен догнать (в ROADMAP это O-5, приоритет поднят).
- **Мелкое расхождение типов исключений в `LexerTypeArray`-эквиваленте:** `src/lexer.ts:100` бросает `LexerException('Unknown LexerType ...')`, PHP-зеркало `php/src/LexerTypeArray.php:83` — `MSLangException`. Текст совпадает, тип — подкласс; префикса координат нет в обоих.
- **Мусор в коде:** устаревший doc-комментарий с упоминанием несуществующего `ParserException` (`src/parser.ts:1878`); маркер `// @TODO` на присваивании `_isPassedByReference` (`src/functionparameter.ts:17`), которого нет в PHP-эталоне.
- **Ссылки на строки в старой карте устарели** после рефакторинга Тира 5 (перенос `ContextInterpreter` и др.): например, O-2 указывал `interpreter.ts:3690` — код теперь в `contextinterpreter.ts:355-356`; O-5 указывал `parser.ts:339,1166,1443` — актуально `346, 1226, 1507`. В новой редакции ROADMAP ссылки исправлены.
- **Структурный огрех прежнего ROADMAP** (исправлен актуализацией): коммит `0bfaa9b` вставил раздел 8 между пунктами 5 и 6 раздела 7 — пункт «6. Дисциплина направления» оказался последней строкой файла после чужого раздела, без завершающего перевода строки.
- **План «работа с данными» не отражён в репозитории как документ.** Код уже есть (`src/jsonfunctions.ts`, `src/stackvariableplainobject.ts`, `src/numberfunctions.ts`, `src/base64functions.ts`, `src/hashfunctions.ts`, `src/urlfunctions.ts`, `src/objectfunctions.ts`, `src/stackvariableenv.ts`), но плановый документ по дальнейшим тирам фич живёт вне репозиториев. Если работа продолжится — стоит завести его рядом с ROADMAP.

## 4. Находки при реализации стрелочных функций (дополнено 2026-07-02, вечер)

### 4.1. Исправленный баг эталона: функция-значение копировалась через хост-обёртку
`let f = function(x) { ... }; f(21)` падал в ОБОИХ движках с `Call to undefined method StackVariableUserFunction::getRequiredCount()`: `varDeclFinishHandler` копировал «примитивы» через `createVariable(vtFunction, ...)`, а фабрика для vtFunction создаёт `StackVariableFunction` — обёртку для хост-функций. Тот же изъян был в связывании параметров (`function ap(fn, v) { return fn(v); }` — падал) и в rest-наборе. Баг был невидим, потому что документация и тесты использовали только присваивание без объявления (`x = function() {...}`), а колбэки массивных методов идут отдельным путём. Исправлено PHP-first в трёх местах (функции — ссылочный тип, как объекты/массивы и как уже было в `makeConstCopy`): `varDeclFinishHandler`, связывание параметров, rest-набор; зеркально в TS. Закреплено парой тестов `096_FunctionValueByReference`.

### 4.2. Попутно исправлено: `['k' => (выражение)]` не парсилось
Гейт ветки `ltLPar` не включал `ntArrayPushSeparatorKey` в список допустимых предшественников — в отличие от соседних гейтов литералов (`ltNumeric`/`ltString`/`ltBracketOpen`/`ltStartCode`). Теперь `['k' => (1 + 2)]` и `['fn' => (x => x + 1)]` работают в обоих движках (тест в `096_ArrowArrayKeys`).

### 4.3. Существующие ограничения, обнаруженные прогонами (НЕ чинились — фиксация)
- **Цепочка вызовов результата не поддерживается:** `f(40)(2)` — ошибка разбора `Parse expression failed` (второй `(` после `ntFuncCall` не принимается). Обход: промежуточная переменная. Каррирование стрелок из-за этого пишется как `let g = f(40); g(2);`.
- **`new T().m()` внутри бинарного выражения** (например `"" + new T().m()`) даёт `ContextException: Unknown function "m"`, хотя `return new T().m();` отдельной инструкцией работает.
- **Инструкция не может начинаться с `[`:** `[1,2,3].map(...);` как statement — `Failed parse code, tokenSym: ltBracketOpen` (выражения-инструкции начинаются только с идентификатора/инкремента/this/super).
- Все три ограничения одинаковы в PHP и TS — паритет не нарушен, это кандидаты в будущие фичи парсера.

### 4.4. Семейное расхождение текстов ошибок new-времени
PHP `InterpreterNode` создаётся без позиции токена, TS — с ней, поэтому `"f" is not a constructor` в PHP приходит без префикса `[строка:столбец]`, а в TS — с ним (`[1:47] "f" is not a constructor`). Это касается всех ошибок, бросаемых из finish-узлов `new`. Тесты 096 сверяют сообщение вхождением. Полное выравнивание — потребует носить позицию в PHP `InterpreterNode` (отдельная задача, к P1-7 не относится — там правились конструкторы исключений).

### 4.5. Дизайн-заметки стрелочных функций (для будущих правок)
- Токен `=>` остался `ltArraySeparator` (38) — новых констант не заводилось, mirror-тест не затронут; узел стрелки — `ntFunctionDef` с меткой `nValue2 = 'arrow'` (рядом с существующей `'expr'`).
- Граница совместимости: внутри литерала массива (`ParentNode.nType === ntArrayPush`) `=>` — всегда разделитель ключа; `[x => 1]` — ключ из переменной (страж — замороженный тест `041_AdvancedArrayFeatures`), `[(x) => 1]` — тоже ключ. Стрелка в массиве — только в скобках: `[(x => 1)]`.
- Стрелка распознаётся только в НАЧАЛЕ выражения (`NodeList` пуст): `1 + x => 2` остаётся ошибкой разбора, как в JS.
- `( ... ) =>` распознаётся заглядыванием вперёд по шаблону for-of (`saveState`/`restoreState`, поиск парной `)` со счётчиком глубины).
- Лексический `this` захватывается в `buildUserFunction` вместе с `_currentMethodOwner` (для `super.method()` внутри стрелки) и подставляется в `invokeUserFunction` вместо динамических значений. TDZ конструктора стрелка не наследует (микро-отличие от JS: вызов стрелки, созданной до `super()`, не бросит TDZ-ошибку — сознательное упрощение).

### 4.6. Найдено состязательной проверкой реализации и исправлено
Проверка (4 линзы × ~250 живых прогонов в обоих движках + независимое подтверждение каждой находки) нашла три настоящих дефекта; все исправлены и закрыты тестами:
- **Обрыв транзитивного замыкания (баг эталона).** `let f = a => b => c => a + b + c; …` падал «variable not defined a» в обоих движках: снимок замыкания брал только `_variables`, теряя то, что уже захватила объемлющая функция. До этой ветки дефект был не виден — возврат функции из функции вообще не работал (пункт 4.1). Исправлено PHP-first: снимок сливается с `_currentCapturedScope`, локальные затеняют внешние. Тесты: каррирование трёх уровней (стрелки и function-выражения) в `096_ArrowClosures`.
- **TS-лексер не переносил `_tokenCursor` через `saveState`/`restoreState`** (PHP переносит). Новое заглядывание вперёд у `(` сделало это наблюдаемым: незакрытая скобка падала внутренней ошибкой `token cursor is not initialized.` вместо парс-ошибки, а позиции `[line:col]` у ошибок скобочных выражений сдвигались на последний токен заглядывания. Исправлено в `CodeLexer.saveState`/`restoreState`; заодно точнее стали позиции у существующего заглядывания for-of. Тесты позиций — в `096_ArrowParseErrors`.
- **Крах TS на обрыве текста после `=>`.** `let f = x =>` (конец файла) ронял TS внутренней ошибкой, PHP давал «Parse expression failed». Причина глубже стрелок: PHP держит курсор токена как `?TokenCursor`, а TS-геттер бросал при его отсутствии. `ParseNode` выровнен под PHP-форму (`TokenCursor | undefined`), чтение курсора в теле стрелки — безопасное. Тест — в `096_ArrowParseErrors`.

Отклонено проверкой (не дефект): `super()` внутри стрелки не снимает TDZ конструктора — обратная сторона принятого упрощения «TDZ не пересекает границу стрелки»; PHP и TS ведут себя одинаково, обход тривиален (звать `super()` напрямую).

Отдельная заметка на будущее (существующее, не от этой ветки): передача пользовательской функции в хост-функцию (`Math.abs(fn)`) — PHP роняет сырой PHP `TypeError`, TS молча отдаёт null; со стрелками нарваться стало проще — кандидат в отдельную задачу.
